PV-01 — REPLACE THE SUPERSEDED ODR-60 ALLOCATION ALGEBRA
======================================================================

The current schema uses aggregate counts and permits multi-role declarations.

Replace it with an exact typed item-level allocation witness.

AXIS A — EXACTLY ONE CONTENT FAMILY PER ITEM

Allowed values:

    BOUNDED-SUPPORT
    SCOPE-AND-VERSION
    CONFLICT-AND-RESIDUE
    NOTATION-NEUTRAL-TRANSFER

The candidate design contains exactly:

    24 item slots
    6 slots per content family

AXIS B — EXACTLY ONE ANSWERABILITY ROLE PER ITEM

Allowed values:

    POSITIVE-CONCLUSION
    DELIBERATE-INSUFFICIENCY
    MIXED-BOUNDED-CONTROL

Enforce:

- every item has exactly one answerability role;
- POSITIVE-CONCLUSION and DELIBERATE-INSUFFICIENCY cannot coexist;
- no array or multi-role escape hatch is permitted;
- answerability-role totals are derived from the item rows.

CROSS-CUTTING TAGS

Allowed tags:

    TRAP-BEARING
    SHAM-DESIGNATED
    EASY-BOUNDED-CONTROL
    STRONG-CONCLUSION-CONTROL
    DOMAIN-NATIVE-NON-LANG-A-RENDERABLE

Enforce:

- STRONG-CONCLUSION-CONTROL implies POSITIVE-CONCLUSION;
- exactly two SHAM-DESIGNATED items exist in every family;
- at least one DOMAIN-NATIVE-NON-LANG-A-RENDERABLE item exists in every family;
- the NOTATION-NEUTRAL-TRANSFER family does not satisfy that obligation for the
  other three families;
- at least eight POSITIVE-CONCLUSION items exist overall;
- at least eight DELIBERATE-INSUFFICIENCY items exist overall;
- at least eight TRAP-BEARING items exist overall;
- no duplicated item slot exists;
- all totals are derived from the 24 item rows;
- SHAM designation changes neither family nor answerability role.

PROPOSED APPENDS — REPAIR 0.2 — AWAITING OWNER ADOPTION

APPEND A — COMMISSION CUSTODY. Track the exact bytes of this commission as
evidence: `REPAIR-0.2-COMMISSION` with SHA-256 and byte length. Bind every PV
adjudication and the ODR-60 candidate witness to that digest as their basis.
The candidate allocation payload must carry provenance
`owner-supplied-via-commission` bound to the same digest, so no later reader
can mistake it for a builder default. Faithful-incorporation review of this
repair is impossible without the instruction bytes fixed; the prior tranche did
this for the docket via authorial-input custody and this tranche should do it
for its own charge.

APPEND B — ESCAPED-DEFECT FIXTURES. Encode each PV counterexample as a
permanently tainted regression fixture with a stable ID —
`mutation:pv-01-witness` through `mutation:pv-05-witness` — carrying the exact
witness bytes that defeated the prior validator. The registry carries them
forever. Standing rule for all future tranches: every defect that escapes a
review becomes a permanent named mutant; escapes are promoted to law, not
merely fixed.

APPEND C — VACUOUS-QUANTIFIER AUDIT. Generalize PV-05 to its class. Enumerate
every validator check that quantifies over a collection; declare, per check,
its behavior on the empty collection; any universal check over a collection
whose emptiness would be semantically vacuous must pair with an explicit
nonemptiness rule or a typed empty-state. Add at least one empty-collection
mutant per adjudicated surface beyond transmission — empty rendering set on an
item record, empty `identity_bindings` on a taint record, empty
`subject_sha256s` — each declared, executed, killed. Deliver the enumeration as
tracked evidence: `VACUOUS-QUANTIFIER-CENSUS`.

APPEND D — SYNTHETIC-BYPASS MUTANTS. For PV-03, add mutants that attempt each
transition in synthetic mode with one requirement removed:
`allow_synthetic=True` with null freezer decision; with missing transition
event; with unresolved freezer actor. Each must be killed by the same typed
condition as its real-mode twin, proving synthetic mode shares the gate rather
than shadowing it.

APPEND E — REGISTRY APPEND-ONLY ACROSS TRANCHES. Mutation IDs are immutable; no
renumbering of the existing 45; the expanded registry records the predecessor
registry's digest; `declared_unexecuted` and `undeclared_executed` remain
first-class and must be empty. Evidence comparability across tranches depends
on stable mutant identity.

APPEND F — RUNTIME IDENTITY IN RUN RECORDS. Each fresh verification-run record
must bind interpreter version, jsonschema version, platform triple, and exact
command bytes. If both fresh runs share one environment, the record states
single-environment scope explicitly rather than implying portability. This
closes PA-N4 and prevents FI-17's recurrence in this tranche's own evidence.

APPEND G — CANONICALIZATION CONTRACT AND GOLDEN VECTOR. State the
canonical-bytes rule in prose in the record contract: UTF-8, sorted keys,
compact separators, `ensure_ascii` disabled, trailing line feed, the two
derived members excluded. Commit one golden-vector fixture — a minimal record,
its exact canonical bytes, and its digest — so an external replayer can
conformance-test its own canonicalization before touching real records. Add a
mutant: golden vector with one altered byte fails digest validation. This
closes PA-N1; an independent recomputation failed all 44 lineage digests over
one undocumented newline, and the next reviewer should not pay that toll.

APPEND H — DERIVED-TOTALS MUTANT. The ODR-60 witness must not store totals; all
cardinalities are recomputed from the 24 rows at validation. If any total is
stored for display, add a mutant: declared total inconsistent with derived
rows, rejected. The number that can drift from its derivation eventually will.
