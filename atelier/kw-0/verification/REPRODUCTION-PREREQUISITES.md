# KW-0 reproduction prerequisites (hard requirements, owner-ruled)

1. **SBCL 2.4.6 exactly.** Hard requirement, not an environment note. Verified divergence: under SBCL 2.2.9 the seven-death generation completes but journals diverge from the reference (three scenarios; preserved at `FAILED-sbcl229-stdout.txt`). This is a genuine portability finding, not an error in either run.
2. **`/tmp` packaging-path collision — PACKAGING DEFECT (code inspection supports defect).** The packaging-edited `reproduce.sh` binds `/tmp/kw`, `/tmp/sbcl-bin`, and `/tmp/sbcl-2.4.6-x86-64-linux/obj/sbcl-home` with unconditional `ln -sfn` and no pre-existence check. If `/tmp/sbcl-2.4.6-x86-64-linux/obj/sbcl-home` already exists as a real directory (e.g., an SBCL binary tarball extracted at its default name — the collision that occurred here), `ln -sfn` silently creates the symlink *inside* it and the harness receives a coreless `SBCL_HOME`; every runner dies at startup (preserved at `FAILED-envcollision-stdout.txt`). Fix owed in any future packaging: fail loudly if the bind targets exist as non-symlinks.

Remaining requirements as declared in the bundle: Linux x86-64, bash, `grep -P`, `md5sum`, Python ≥3.8 stdlib, writable `/tmp` with POSIX fsync semantics, repo checkout at pinned commit `f8842f8c…` (verify `deps/PINNED-COMMIT.txt` hashes byte-for-byte). Reference generation was overlayfs; ext4 reproduced byte-identically here.
4. Run generations **sequentially** — the `/tmp` bindings are global; parallel extractions collide by construction.
